# 电商项目需求梳理

## 一、核心结论

本次沟通实际讨论了两条主线：

1. **ERP / 财务核算底座**：长期目标是重做一套电商 ERP，用于管理商品、采购、付款、成本分摊、回款、税费和利润核算，核心能力是把所有成本尽可能归集到具体 SKU、批次，最终细到单件商品。
2. **运营赚钱工具**：短期更重要的是通过数据分析、补货决策、定价策略、竞品分析和异常提醒，提升运营效率和利润。当前业务在亏钱或低效运行，因此优先级高于完整 ERP。

短期建议不是先做一个大而全系统，而是先做若干个 0.1 / 0.2 版本的小工具，把数据底座、周报、补货测算、异常检测、竞品分析逐步跑起来。

## 二、业务背景

当前业务是本土电商模式：国内公司采购商品，发往俄罗斯、白俄罗斯等当地公司或店铺，在当地平台销售。不是传统跨境电商的一单一发模式，而是提前囤货到当地仓库，再通过本土店铺销售。

目前存在的问题：

- 商品、采购、付款、成本、库存、销售、回款大量依赖表格。
- 很多费用已经发生，但无法准确知道摊到了哪些商品上。
- 不清楚每个 SKU、每个批次，甚至每件商品真实赚了多少钱。
- 国外店铺销售后是否回款、回了哪些货款，目前不够清晰。
- 平台费用、广告费、物流惩罚、税费等影响利润，但缺少系统化归因。
- 运营人员管理多个店铺和大量商品链接，靠人工容易遗漏。
- 采购倾向于一次买多拿低价，但这可能牺牲资金周转和年化回报率。

## 三、长期目标：ERP / 财务核算底座

### 1. 商品与 SKU 体系

需要建立完整商品资料体系：

- 三级类目：一级类目、二级类目、三级类目。
- SKU 属性：颜色、规格、尺寸、重量、体积、型号、品质、特点等。
- SKU 与实体货物关系：一个 SKU 可有多个批次，每个批次包含多件实体商品。
- 单件商品 ID：长期希望每件商品都有唯一 ID，甚至可生成二维码。

目标不是只算 SKU 均价，而是能追踪到每一件实体商品在生命周期内发生的成本和收益。

### 2. 国内采购与付款管理

国内端需要覆盖：

- 采购合同。
- 供应商 / 工厂。
- 采购数量、单价、首付款、尾款。
- 发票与付款申请。
- 财务付款审批，至少两级审核。
- 付款后成本锁定，审核通过后的真实成本不可随意修改。

当前已有类似汽车业务的审批和付款逻辑，可考虑复用思路。

### 3. 成本分摊

系统核心是分摊。所有费用最终都要尽可能分摊到 SKU、批次或单件商品上。

已提到的成本类型：

- 商品采购成本。
- 国内付款成本。
- 国际物流费。
- 清关、税费。
- 认证费。
- 仓储费。
- 国内或海外中转物流费。
- 平台佣金。
- 广告费。
- 平台会员费。
- 物流惩罚费。
- 客诉赔付。
- 退货损失。
- 资金利息。

分摊方式不要求一开始高度自动化，早期可以强人工录入和分配，但要保证财务付款金额与商品成本归集一致。

典型规则：

- 采购款：人工选择分摊到对应 SKU / 批次。
- 国际物流费：可按体积、重量或人工指定分摊。
- 认证费：可能挂在 SKU 上，也可能由首批、前 1000 件或每件固定金额承担。
- 平台费用：部分可自动按平台账单摊到订单 / 商品，部分需要规则确认。
- 会员费等固定费用：是否摊到当月销售商品，需要业务定规则。

### 4. 国外销售与财务核算

国外端以店铺为核心，需要拉取平台账单和运营数据。

关键对象：

- 国家。
- 公司。
- 平台。
- 店铺。
- 仓库。
- SKU。
- 商品链接。
- 订单。
- 平台账单。
- 税制和佣金规则。

不同国家、不同公司、不同平台、不同店铺的税制、平台佣金、物流费用和结算规则都可能不同。同一件商品在不同店铺销售，最终利润可能差异很大。

### 5. 回款与闭环

长期需要打通完整闭环：

采购付款 → 发货出口 → 到海外仓 → 店铺销售 → 平台结算 → 国外公司回款 → 国内财务核算 → 退税 / 利润确认。

回款也需要能拆到具体 SKU、批次或商品上，否则无法知道哪些货已完成资金闭环。

## 四、短期目标：运营赚钱工具

短期重点不是先把 ERP 做完整，而是先解决「怎么赚钱」。

### 1. 周报 / 日报与利润看板

需要先把平台运营数据、财务数据和成本数据汇总，形成每周或每天可看的结果。

看板需要回答：

- 上周每个 SKU 卖了多少？
- 每个店铺卖了多少？
- 实际收入是多少？
- 平台扣了哪些费用？
- 税后净利是多少？
- 单品回报率是多少？
- 是否达到预期？
- 是否需要补货、降价、清库或加广告？

### 2. 销售漏斗与财务漏斗

销售漏斗：

- 曝光。
- 浏览。
- 点击。
- 加购。
- 下单。
- 签收。
- 退货。

财务漏斗：

- 订单成交。
- 签收确认。
- 平台出账单。
- 平台扣费。
- 平台结算。
- 税后净收入。

销售数据和财务数据需要按天、按周、按 SKU、按店铺聚合。

### 3. 补货决策工具

补货是当前最重要的决策场景之一。

需要综合：

- 当前库存。
- 每周 / 每月动销。
- 采购周期。
- 生产周期。
- 国际物流周期。
- 安全库存。
- 仓储费。
- 资金占用成本。
- 不同采购数量对应的采购单价。
- 年化回报率。
- 旺季 / 淡季。

目标不是简单判断「卖得好就补」，而是计算补多少、多久补一次、是否值得继续做。

关键观点：

- 一次买更多虽然单价更低，但可能压资金、降年化回报。
- 每月补货可能比三个月补一次更优，但会增加操作成本。
- 长尾品只要回报率足够，也值得做，不应只看绝对利润。
- 爆款和长尾品需要不同策略。

### 4. SKU 分级：S / A / B / C

需要基于预测和实际经营结果，对 SKU 做分级。

可能的策略：

- S 级：高回报、高潜力，重点投入广告、图片、竞品分析、库存保障。
- A 级：稳定赚钱，持续补货，控制库存和资金效率。
- B 级：小额稳定利润，低维护成本运行。
- C 级：亏损或滞销，进入清库流程。

C 级商品需要有明确规则：

- 是否所有店铺同时上架清库。
- 是否限制广告费。
- 是否调价到接近成本。
- 多久清完才算合格。
- 清库亏损上限是多少。

### 5. 运营异常检测

系统需要帮助运营发现人工容易漏掉的问题。

已出现或讨论的异常：

- 商品被平台自动加入清库活动，按极低价格卖出，导致亏损。
- 滞销品没有上到所有店铺清库。
- 热销品断货，没有及时补货。
- 应该空运补货的高利润商品，因舍不得物流费导致错失利润。
- 同一个 SKU 在不同店铺利润差异过大。
- 某个店铺低价低利润占用了库存，影响利润最大化。
- 广告在无库存或库存不足时继续烧钱。
- 某区域销售好，但没有提前分仓，导致物流惩罚或排名受影响。

### 6. 竞品与选品分析

需要利用第三方工具、平台数据、爬虫或人工辅助采集竞品数据。

可采集内容：

- 竞品价格。
- 月销量 / 日销量。
- 销售额。
- 排名。
- 转化率。
- 主图、详情图、标题、卖点。
- 参数。
- 评论。
- 关键词。
- 活动和促销方式。

应用场景：

- 新品开发前预测销量。
- 判断乐观 / 中性 / 悲观销量区间。
- 对比自身商品为何转化率低。
- 提炼竞品卖点。
- 监控竞品调价。
- 辅助定价和广告策略。

## 五、数据与技术方向

### 1. 数据底座优先

需要有专门任务定时拉取平台数据，存入本地数据库，并加工成可分析的数据。

原则：

- 分析工具尽量使用本地数据，不要每次实时调用平台 API。
- 原始数据要长期保留，方便规则变更后重算。
- 平台 API 可能限流，且字段含义不一定清晰，需要建立自己的解释规则。
- 对未知情况不能只做二选一判断，需要保留「未知 / 待确认」状态。

### 2. 工具化开发

短期更适合拆成多个小工具：

- 平台数据同步工具。
- 成本和利润周报。
- SKU 回报率计算器。
- 补货测算工具。
- 商品分级工具。
- 运营异常提醒。
- 竞品分析工具。
- 清库策略工具。
- 多店铺价格 / 上架辅助工具。

不建议一开始就做一个完整大系统，容易需求不清、周期过长、上线后难以迭代。

### 3. AI 使用方向

AI 更适合做：

- 平台规则整理。
- 账单解析。
- 竞品卖点提炼。
- 评论总结。
- 运营报告草稿。
- 异常原因分析。
- 多维度数据分析辅助。
- 生成调价、补货、广告建议。

但算钱、成本、税费、回款等强财务场景必须谨慎，不能依赖 AI 猜测。

## 六、待确认问题

### 1. ERP 范围

- 第一版 ERP 是否只做国内采购、付款、SKU、成本分摊？
- 是否第一版就要做到单件商品 ID，还是先做到 SKU + 批次？
- 国内端和国外端是否拆成两个项目？
- 多国家、多平台、多店铺的规则抽象，第一版需要做到什么程度？

### 2. 成本分摊规则

- 采购款分摊到 SKU、批次还是单件？
- 物流费默认按体积、重量、货值，还是人工指定？
- 认证费默认如何摊：SKU 挂账、首批承担、前 N 件承担，还是每件固定金额？
- 平台会员费、固定服务费是否摊到商品？
- 仓储费和资金利息是否第一版纳入？
- 财务审核后哪些成本字段不可修改？

### 3. 平台账单与财务核算

- 平台账单是否能稳定关联到订单 / 商品 / SKU？
- 每个平台每日、每周、每月分别能拿到哪些账单？
- 不可分摊费用如何处理？
- 当地财务报表与系统报表按什么口径核对？
- 税费规则由谁维护，是否需要规则版本？

### 4. 回款规则

- 国外公司回款是否必须拆到 SKU / 批次 / 商品？
- 一笔回款如何对应多个销售周期或多个 SKU？
- 回款未拆清时是否允许入账？
- 回款与退税是否第一阶段纳入？

### 5. 补货与回报率模型

- 年化回报率公式最终采用哪一版？
- 安全库存默认按几个月计算？
- 采购周期默认是 3 个月，是否按供应商或品类配置？
- 旺季系数如何维护？
- 采购量与采购单价的阶梯价格如何录入？
- 不同店铺、不同仓库的库存是否独立建模？

### 6. SKU 分级规则

- S / A / B / C 的阈值是什么？
- 按年化回报率、绝对利润、动销速度，还是组合评分？
- C 级清库的最大亏损边界是多少？
- 长尾品是否允许低绝对利润但高回报率继续保留？

### 7. 数据采集

- 当前已经接入哪些平台 API？
- 哪些数据已经能稳定拉取？
- 第三方工具数据是否允许导出，格式是什么？
- 爬虫是否允许做，风险和账号策略如何控制？
- 原始数据存储周期和重算机制如何设计？

### 8. 项目协作

- 技术栈选型：Ruby、Python，还是混合？
- 核心系统和脚本工具如何分层？
- 文档驱动开发的文档格式、维护责任和更新流程是什么？
- 测试规范如何定，哪些模块必须有测试？
- 多人协作时模块边界如何划分？
- 生产数据上线后如何做迁移、备份和回滚？

## 七、建议的开发路线

### 第 0 阶段：需求和架构梳理

目标：先画清楚全局架构和数据流，不急着写大系统。

产出：

- 国内采购到海外销售的完整流程图。
- 核心对象关系图：SKU、批次、单件商品、采购单、付款、费用、店铺、订单、账单。
- 国内端、国外端、工具端的边界划分。
- 第一版数据字典。
- 第一版成本分摊规则文档。

### 第 1 阶段：数据底座和周报

目标：先知道每周到底赚没赚钱。

产出：

- 平台数据定时拉取。
- 原始数据入库。
- SKU / 店铺维度销售周报。
- 平台费用和税后净利汇总。
- 基础利润看板。

### 第 2 阶段：补货与回报率工具

目标：让业务能判断一个品是否继续做、补多少、多久补一次。

产出：

- SKU 回报率计算器。
- 补货测算工具。
- 安全库存建议。
- 不同采购数量和价格下的回报率对比。
- 历史预测与实际结果对比。

### 第 3 阶段：异常检测和运营提醒

目标：让系统主动找问题，降低运营遗漏。

产出：

- 亏损销售提醒。
- 异常低价销售提醒。
- 断货 / 快断货提醒。
- 广告无效消耗提醒。
- 滞销库存提醒。
- 清库未覆盖全部店铺提醒。
- 店铺利润差异提醒。

### 第 4 阶段：ERP 最小可用版

目标：把国内采购、付款、成本分摊先线上化，减少表格依赖。

产出：

- SKU 和批次管理。
- 采购合同管理。
- 付款申请和审批。
- 采购款分摊。
- 物流费 / 认证费等费用录入与分摊。
- 成本锁定和追溯。

### 第 5 阶段：竞品与选品工具

目标：提升选品和运营策略能力。

产出：

- 第三方竞品数据导入。
- 重点竞品跟踪。
- 竞品价格和销量变化监控。
- 主图、卖点、评论分析。
- 新品销量预测辅助。

## 八、优先级建议

### P0：现在最应该先做

- 平台数据本地化同步。
- 每周 SKU / 店铺利润周报。
- SKU 回报率和补货测算工具。
- 断货、亏损销售、异常低价等基础异常提醒。

### P1：近期应该启动

- SKU 分级规则。
- 清库策略工具。
- 竞品数据导入和基础分析。
- ERP 最小可用版的需求文档和数据模型。

### P2：中长期建设

- 完整 ERP。
- 单件商品级成本追踪。
- 多国家、多平台规则引擎。
- 回款、退税、当地财务报表核对。
- AI 多维度运营决策编排。

## 九、关键风险

- 需求仍在探索中，不能过早做重系统。
- 平台 API 字段含义可能不完整，需要人工验证和规则沉淀。
- AI 可以辅助分析，但财务核算不能依赖 AI 猜测。
- 如果没有原始数据存档，后续规则调整会很难重算。
- 数据迁移和表结构频繁变更会带来生产风险。
- 如果文档不持续更新，多人协作会很快失控。
- 运营工具如果只输出「正确废话」，不能落地执行，价值会很低。

## 十、下一步建议

1. 先确认第一阶段目标：是否以「周报 + 补货测算 + 异常提醒」作为 3 个月内第一主线。
2. 把 ERP 降级为并行调研和最小版设计，不阻塞赚钱工具。
3. 拉出现有可用数据清单：平台 API、第三方工具、人工表格、采购数据、库存数据。
4. 先选 3 个典型 SKU 做样板：爆款、腰部品、滞销品。
5. 用样板 SKU 跑通利润核算、补货测算、异常检测和竞品分析。
6. 同步制定项目协作规范：文档结构、数据字典、测试边界、代码模块划分。
