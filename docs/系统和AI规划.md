## 基础数据，尽量提供精准数据，可验证，意义明确
1. 不要出现任何让AI写SQL的情况，SQL仅供初步研究探索使用，不具备通用和后续的价值
2. 将所有所需的基础数据聚合，变成service，用MCP或者API提供到客户端（YClaw），后续service的数据直接供后端AI使用
3. 数据的准确性：确保API、MCP提供的数据是准确的，有地方能对这些数据进行查看和校验。
4. 数据的意义明确：不要提供无意义的数据给到AI，接口返回的数据需要有明确的意义，可以写一段描述文字跟随数据一起返回

## 提供规则，怎么提供？提供什么？什么时候参考？-- gbrain
1. 平台规则：Ozon/WB平台的重要规则，包括费用计算，各地仓库系数，短期、临时的营销、季节性的特性规则，需要提供文档资料来源，由AI定期检测扫描（一周一次）变更后写入知识库并发送通知站内信，由AI判断是否重要。
2. 例如分级策略这种硬性的规则，尽量使用代码计算并写死
3. 运营规则，营销思路等，应当融入skill或者agent
4. 一些不太重要或者参考资料直接一次性录入知识库即可，转成markdown让专人审核录入
5. 什么时候参考知识库：需要在Agent里声明（像在这个岗位的员工一样）做方案前先看看相关规则

## 如何架构Agent和skill
1. 基础概念：Agent和skill的拆分，尽量让Agent处理一个方向的事，越垂直效果越好，其中牵扯到不能有误差汇总计算的情况需要让AI写脚本计算而不是自己估算。Skill则是能支撑这个Agent能完成这个方向工作的数据支撑、规则指引。
我们可以先约定，例如分仓就做一个Agent出来，由1~3个skill提供所需数据。skill尽量做更细致的事，例如库存skill专门提供库存概况和各平台各地区仓库的库存分布；仓库分布skill则列出所有仓库和地区以及一些仓库特性；
这样就能保证后续能够方便组合skill来完成Agent的目标。

2. 工作流：<团队模式> 由leader制定目标，多Agent编排成工作流或者并行给出结论，最终leader汇总或者交给最终Agent汇总

3. 怎么做？一句话：SOP + 数据支撑
当前最快速就是有SOP之后，通过浏览器打开指定页面收集准确的数据，快速调试可行性，调试提示词，需要关注的规则和明确预期最终落地变成一个Agent，外部数据入库，内部数据分类变成一个个Service，组合成若若干个skill
4. 怎么验证调试？先写草稿SOP，明确所需数据，可以先结合系统模拟数据直接以对话的形式补充（不管是模拟的数据还是真实的数据，我们都必须有明确的结果预期），这样反复调试agent直到Agent的工作流程和结果都是符合我们预期的。
5. 胡出SOP，夏写Agent，皮&胡提供数据和预期处理流程和结果，夏调试提示词并完善skill
