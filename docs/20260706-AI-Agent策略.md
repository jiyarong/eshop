# AI 经营策略

本文基于《SKU 分级经营管理规范》和《SKU 分级运营 SOP》重新梳理 AI Agent 的建设方向。

AI 经营系统的核心目标不是做一个可以随便聊天的“大模型助手”，而是把 SKU 经营管理中的判断、监控、复盘和预警流程系统化。所有 AI 能力都应围绕两个经营标签展开：

- `SKU Grade`：决定资源投入优先级，即这个 SKU 值不值得投入人、货、钱、广告和供应链资源。
- `SKU Stage`：决定当前经营目标，即这个 SKU 现在最该解决什么主要矛盾。

因此，AI 的第一性原则是：**先帮助运营判断 Grade / Stage / 主要矛盾，再给出下一步动作建议。**

## 一、总体定位

### 1. AI 不是自动经营者

AI 不应直接替代运营人员做最终经营决策，尤其不能自动决定大额补货、清库、涨价、停投广告等高风险动作。

AI 应承担以下工作：

1. 整理 SKU 当前经营上下文。
2. 按 SOP 顺序完成初步诊断。
3. 判断是否存在异常或分级调整信号。
4. 生成可复盘的经营分析报告。
5. 建议下一步动作、观察周期和验证指标。
6. 在动作执行后自动跟踪结果。

最终确认仍由运营人员负责。

### 2. AI 不直接从原始表里猜结论

经营指标必须先由 Rails 服务或规则引擎确定性计算，再交给 AI 分析。AI 不负责财务核算、利润归因、订单归属等确定性计算。

推荐数据流：

```text
平台原始数据 / ERP 业务数据
-> Rails 指标聚合服务
-> SKU 经营上下文 JSON
-> 规则判断 Grade / Stage / 异常信号
-> AI 生成解释、归因、建议和复盘报告
-> 人工确认
-> 运营动作 / 监控任务 / 后续报告
```

### 3. AI 的分析顺序必须遵循 SOP

AI 经营分析不能看到广告变差就直接说“广告有问题”。分析顺序必须固定：

```text
库存 -> 商品竞争力 -> 广告 -> 利润 -> Grade / Stage 调整建议
```

库存和商品竞争力是经营基础，广告是获取流量的工具，利润是经营结果。AI 的价值在于帮助运营找到当前主要矛盾，而不是孤立解释单个指标。

## 二、建议构建几个 Agent

完整体系建议构建 **6 个 Agent / 能力模块**。其中 MVP 先做前 4 个，后 2 个作为第二阶段。

这里的 Agent 不一定都要是独立的聊天机器人，也可以是后台服务、定时任务、规则引擎和 AI 分析器的组合。关键是职责边界清楚、输入输出稳定。

| Agent | 阶段 | 核心职责 | 是否需要 AI |
| --- | --- | --- | --- |
| 1. 经营事件 Agent | MVP | 记录运营动作，形成可追踪事件 | 部分需要 |
| 2. SKU 经营画像 Agent | MVP | 汇总 SKU 的经营指标包 | 不一定需要 |
| 3. 分级诊断 Agent | MVP | 判断 Grade / Stage / 红黄绿健康状态 | 规则为主，AI 辅助解释 |
| 4. 监控复盘 Agent | MVP | 根据动作创建观察任务，到期生成对比报告 | 需要 |
| 5. 异常预警 Agent | 第二阶段 | 主动发现断货、亏损、广告浪费、清库异常等问题 | 规则为主，AI 辅助 |
| 6. 运营 Copilot Agent | 第二阶段 | 在 SKU 详情页回答原因、建议和历史复盘问题 | 需要 |

MVP 不建议一开始做“自动调价”“自动开关广告”“自动补货”。先把经营动作记录、分级诊断、监控复盘跑通，才能让 AI 的建议有依据、有闭环。

## 三、Agent 设计

### 1. 经营事件 Agent

经营事件 Agent 负责把运营动作统一记录下来。没有动作记录，就无法判断后续数据变化是市场自然变化，还是自身运营动作造成的。

典型事件：

- 调整价格。
- 调整广告预算。
- 开启或关闭广告。
- 调整广告出价、关键词、否词。
- 优化主图、标题、详情页。
- 参加或退出平台活动。
- 新增官方仓分仓。
- 补货、采购、清库。
- 修改 SKU Grade 或 SKU Stage。

事件记录至少包括：

- `event_type`：动作类型。
- `sku_code`：内部 SKU。
- `platform` / `store_id`：平台与店铺。
- `listing_id` / `product_id` / `platform_sku_id`：平台商品标识。
- `actor_id`：操作人。
- `happened_at`：动作发生时间。
- `before_payload`：动作前状态。
- `after_payload`：动作后状态。
- `intent`：本次动作目的。
- `expected_effect`：预期改善指标。
- `observe_days`：计划观察周期。
- `tags`：人工标签，例如“测试涨价”“清库”“验证主图”。

已有 `ec_operation_logs` 可以作为审计基础，但它偏 ERP 字段变更日志，不足以承载完整经营语义。建议新增业务事件层，例如 `ec_operation_events`。

### 2. SKU 经营画像 Agent

SKU 经营画像 Agent 负责为每个 SKU 生成统一的经营上下文。它本质上更像数据聚合服务，不一定需要大模型参与。

输出应是结构化 JSON，而不是自然语言。

画像至少包括：

- SKU 基础信息：SPU、品类、负责人、平台绑定、店铺绑定。
- 当前 Grade / Stage。
- 近 1 周、4 周、8 周销售趋势。
- SKU 汇总利润、Listing 利润。
- 广告费用、广告占比、广告 ROI。
- 流量漏斗：曝光、点击、加购、下单、转化。
- 库存结构：官方仓、FBS、本地仓、在途、采购待到。
- 库存可售天数、预计断货日期、积压风险。
- 退货、取消、差评等风险指标。
- 最近经营事件和未完成监控任务。
- 已知数据缺口。

画像粒度要同时支持三层业务模型：

- SPU：用于产品线和开发决策。
- SKU：用于资金、库存、Grade / Stage 判断。
- Listing：用于日常运营诊断和广告分析。

订单归属必须通过 `ec_sku_products` 绑定关系确认，不能用 `ec_order_items.sku_code` 兜底作为统计依据。

### 3. 分级诊断 Agent

分级诊断 Agent 是 AI 经营系统的核心。它不只是给 SKU 打红黄绿灯，而是要判断：

1. 当前 Grade 是否合理。
2. 当前 Stage 是否合理。
3. 当前主要矛盾是什么。
4. 本周是否需要调整经营策略。

诊断逻辑应先规则化，再由 AI 解释。

#### Grade 诊断

Grade 代表资源投入优先级，应以 SKU 维度聚合判断。

核心指标：

- 年化净利。
- 年化 ROI。
- 近 4 周利润趋势。
- 近 4 周销量趋势。
- 库存资金占用。
- 是否具备继续放量空间。

建议规则：

| Grade | 经营含义 | AI 重点判断 |
| --- | --- | --- |
| S | 现金牛或高潜力核心款 | 是否应继续加资源、加分仓、加广告、扩 SPU |
| A | 主力利润款 | 是否有机会突破到 S，是否存在断货或利润下滑风险 |
| B | 长尾或待验证款 | 是否具备成长为 A 的条件，还是应低成本经营 |
| C | 清库或低价值款 | 是否应停止投入，是否需要加速清库 |

#### Stage 诊断

Stage 代表当前主要经营目标。

| Stage | 当前目标 | AI 重点判断 |
| --- | --- | --- |
| NEW | 验证市场需求 | 是否完成基础分仓、销售漏斗是否成立、是否具备稳定成交能力 |
| GRW | 探索经营上限 | 是否应增加分仓、广告、Listing 优化和竞品分析 |
| MAT | 稳定利润 | 是否维持生态位，是否在销量、广告和利润之间达到平衡 |
| CLR | 回收资金 | 是否按清库目标推进，是否仍在浪费广告和运营资源 |

#### 健康状态

红黄绿灯用于列表页快速筛选，不替代 Grade / Stage。

| 状态 | 含义 | 示例 |
| --- | --- | --- |
| 绿色 | 当前经营符合 Grade / Stage 目标 | S-GRW 销量增长、库存健康、广告占比可控 |
| 黄色 | 存在单项风险，需要本周关注 | A-MAT 利润下降、B-GRW 转化下滑、库存低于安全线 |
| 红色 | 经营目标明显受阻，需要立即处理 | S/A 断货风险、持续亏损、广告烧钱无转化、C-CLR 清库停滞 |

AI 输出不应只写“健康/不健康”，必须给出主要矛盾：

- 库存问题。
- 商品竞争力问题。
- 广告效率问题。
- 价格竞争力问题。
- 平台差异问题。
- 清库执行问题。
- 数据缺口或归因不充分。

### 4. 监控复盘 Agent

监控复盘 Agent 负责把“操作-观察-复盘”变成闭环。

当经营事件发生后，系统应自动创建监控任务。

示例：

| 动作 | 默认观察周期 | 对比方式 | 重点指标 |
| --- | --- | --- | --- |
| 调价 | 3 到 7 天 | 动作后 vs 动作前同等周期 | 销量、转化率、利润、客单价 |
| 增加广告预算 | 7 天 | 本周 vs 上周 | 曝光、点击、订单、广告占比、自然流量 |
| 降低广告预算 | 7 天 | 本周 vs 上周 | 自然排名、自然流量、利润、销量 |
| 主图优化 | 7 天 | 动作后 vs 动作前 | CTR、曝光、加购率、转化率 |
| 新增分仓 | 7 到 21 天 | 分仓区域订单变化 | 区域销量、本地化指数、可售天数 |
| 清库降价 | 7 天 | 清库目标进度 | 动销、亏损边界、预计清完日期 |

监控报告必须结构化：

- 本次动作是什么。
- 当时为什么做。
- 预期改善什么。
- 观察周期是什么。
- 动作前后核心指标如何变化。
- 是否达到预期。
- 主要原因判断。
- 副作用或风险。
- 建议继续、回退、加码、停止还是延长观察。
- 是否建议调整 Grade / Stage。

这部分是 AI 最适合发挥价值的地方，因为它需要把多维指标翻译成运营人员能执行的结论。

### 5. 异常预警 Agent

异常预警 Agent 用于降低运营遗漏，优先规则化。

第一阶段可监控：

- S/A SKU 快断货。
- NEW SKU 上架后未完成基础分仓。
- GRW SKU 库存或广告不足，影响成长验证。
- MAT SKU 利润连续下降。
- CLR SKU 清库速度不足，预计无法在目标周期内清完。
- 广告费用占比异常升高。
- 广告有点击无成交。
- 销量下降但库存、价格、广告没有明显变化。
- SKU 未在应经营的平台或店铺上架。
- 清库 SKU 没有覆盖全部可清库店铺。
- 平台异常低价销售造成亏损。

预警输出应包含：

- 预警等级。
- 影响 SKU / Listing。
- 触发规则。
- 相关指标。
- 建议动作。
- 是否需要创建经营任务。

### 6. 运营 Copilot Agent

运营 Copilot 用于 SKU 详情页或经营分析页，面向运营人员交互。

它应该回答的问题包括：

- 这个 SKU 本周为什么变差？
- 当前主要矛盾是什么？
- 这个 SKU 应不应该从 NEW 进入 GRW？
- B-GRW 有没有机会升级为 A？
- A-MAT 最近利润下降，是广告问题还是价格问题？
- C-CLR 为什么清不动？
- 上周调价有没有效果？
- 这个 SKU 下一步应该先调价、调广告、补货还是优化 Listing？

Copilot 的回答必须引用结构化经营上下文和历史事件，不能凭空建议。

## 五、页面与产品形态

### 1. SKU 列表页

列表页应从“管理列表”升级为“经营驾驶舱”。

建议展示：

- SKU Code / 名称 / 品类 / 负责人。
- Grade / Stage。
- 健康状态：红黄绿。
- 本周主要矛盾。
- 近 4 周销量、利润、广告占比、库存可售天数。
- 是否有未完成监控任务。
- 是否有 AI 建议调整 Grade / Stage。

列表页的目标不是展示所有数据，而是帮助运营快速排序：

1. 哪些 S/A SKU 有风险。
2. 哪些 NEW SKU 需要尽快验证。
3. 哪些 GRW SKU 值得加资源。
4. 哪些 MAT SKU 利润正在衰退。
5. 哪些 CLR SKU 清库执行不到位。

### 2. SKU 详情页

详情页应围绕“动作和数据的关系”展开。

建议模块：

- SKU 经营概览。
- Grade / Stage 当前状态和历史变更。
- 雷达图：财务、销售、库存、增长、广告、竞争力。
- 经营事件时间线。
- 关键指标趋势图。
- 动作前后对比报告。
- Listing 维度诊断。
- AI 经营建议。
- 人工复盘记录。

最重要的是把运营动作叠加到趋势图上，让运营看到“做了什么动作，后来数据发生了什么变化”。

### 3. 周度经营分析页

周度页服务于周一报表和周二经营会议。

建议输出：

- S 级 SKU 重点看板。
- A 级 SKU 风险看板。
- NEW SKU 验证进度。
- GRW SKU 成长机会。
- MAT SKU 利润波动。
- CLR SKU 清库进度。
- Grade / Stage 建议调整清单。
- 本周需要讨论的异常 SKU。
- 上周经营动作复盘结果。

## 六、MVP 落地顺序

### 第 1 步：补经营事件层

目标：先让系统知道运营做了什么。

产出：

- `ec_operation_events`。
- 人工记录入口。
- 从已有 `ec_operation_logs` 转换部分事件。
- 支持记录动作目的、预期指标和观察周期。

### 第 2 步：做 SKU 经营画像

目标：让每个 SKU 都能生成统一指标包。

产出：

- `SkuBusinessProfileQuery` 或类似服务。
- 输出 SKU / Listing / SPU 三层数据。
- 明确数据缺口。
- 保证订单归属遵循 `ec_sku_products` 绑定关系。

### 第 3 步：做分级诊断

目标：让系统能给出 Grade / Stage / 健康状态建议。

产出：

- Grade 建议规则。
- Stage 建议规则。
- 红黄绿健康状态。
- 主要矛盾分类。
- 人工确认入口。

### 第 4 步：做监控复盘

目标：把运营动作变成可验证闭环。

产出：

- `ec_monitoring_tasks`。
- 动作后自动创建观察任务。
- 到期生成 AI 复盘报告。
- 报告可被人工确认、修改和沉淀。

### 第 5 步：做异常预警

目标：让系统主动找问题。

产出：

- 断货预警。
- 亏损预警。
- 广告浪费预警。
- 清库异常预警。
- Grade / Stage 异常信号。

### 第 6 步：做运营 Copilot

目标：让运营可以围绕 SKU 追问。

产出：

- SKU 详情页 AI 问答。
- 周度经营会议问答。
- 历史动作与结果追踪。
- 经营经验沉淀。

## 七、AI 输出规范

AI 报告必须可执行，不能输出正确但无用的废话。

每份报告至少包含：

1. 当前结论。
2. 当前 Grade / Stage 是否合理。
3. 本周主要矛盾。
4. 关键证据。
5. 建议动作。
6. 预期改善指标。
7. 建议观察周期。
8. 风险和副作用。
9. 需要人工确认的问题。
10. 是否建议调整 Grade / Stage。

示例输出结构：

```json
{
  "summary": "该 SKU 当前更像 B-GRW，但成长验证不足，主要矛盾是库存分仓不足而不是广告效率。",
  "grade_suggestion": "B",
  "stage_suggestion": "GRW",
  "health_status": "yellow",
  "main_constraint": "inventory_distribution",
  "evidence": [
    "近 4 周转化率稳定",
    "广告占比未明显恶化",
    "新增区域无官方仓库存",
    "该区域已有自然订单信号"
  ],
  "recommended_actions": [
    "评估新增官方仓分仓",
    "广告预算保持不变观察一周",

```

## 八、关键约束

1. AI 只做建议，不做最终确认。
2. 财务、利润、ROI 必须由确定性代码计算。
3. Grade / Stage 可由系统建议，但必须人工确认。
4. 每次经营动作必须记录意图和观察周期。
5. 每次监控报告必须能追溯到具体动作和指标。
6. 一次经营动作尽量只调整一个主要变量，避免无法归因。
7. SKU 维度用于经营分级，Listing 维度用于战术诊断，SPU 维度用于产品线规划。
8. C-CLR 原则上不再做增长型分析，AI 应避免建议继续投入资源。
9. S/A SKU 的断货、利润下滑、广告异常应高优先级预警。
10. AI 经营策略必须能沉淀经验，逐步把重复有效的判断固化为规则。

## 九、结论

AI 经营系统应围绕 SKU 分级经营体系建设，而不是围绕聊天能力建设。

建议完整构建 **6 个 Agent / 能力模块**：

1. 经营事件 Agent。
2. SKU 经营画像 Agent。
3. 分级诊断 Agent。
4. 监控复盘 Agent。
5. 异常预警 Agent。
6. 运营 Copilot Agent。

MVP 先做前 4 个：**记录动作、汇总画像、诊断分级、监控复盘**。

这样可以先形成最小经营闭环：

```text
经营动作 -> 数据变化 -> AI 复盘 -> Grade / Stage 建议 -> 下一步动作
```

等这个闭环稳定后，再扩展异常预警和运营 Copilot，最后再考虑自动化调价、自动化广告和自动化补货等高风险能力。
